# ADR 001: Go, SSE, Gemini API を用いた Docker コンテナサービスのローカル再現と技術選定

## ステータス
**提案中 (Proposed)**

## コンテキスト (背景と要件)
将来的に AWS ECS (Elastic Container Service) 上で Docker コンテナを基盤とした Go のバックエンドサービスを本番公開する必要があります。
本番移行をスムーズに行うための前段階として、ローカルの Docker 環境上で以下の要件を満たすバックエンドおよびフロントエンドの検証環境（チュートリアル）を構築します。

1. **REST API**: プロンプトの受信など、標準的なリクエスト処理。
2. **Server-Sent Events (SSE)**: チャットの返答をリアルタイムにフロントエンドへストリーミング配信する。
3. **Gemini API 連携**: 外部の Gemini API からのストリーミング応答を中継（代行）し、SSE でフロントエンドに送信する。
4. **コンテナ化 (Docker)**: 将来的な ECS デプロイメントでそのまま利用可能なセキュアかつ軽量なイメージ設計。

---

## 決定事項 (Decisions)

### 1. Webフレームワークの選定
- **決定**: Go 標準パッケージ (`net/http`) を使用する。
- **理由**: 外部依存関係（Gin, Fiberなど）を排除することで、コンテナサイズを極小化し、セキュリティ上の脆弱性リスクを減らします。Go 標準の `net/http` はパフォーマンスが非常に高く、SSE の実装（`http.ResponseWriter` と `http.Flusher` によるフラッシュ制御）も簡潔に行えます。

### 2. Gemini API 統合
- **決定**: Google 公式 SDK `github.com/google/generative-ai-go/genai` を使用する。
- **理由**: 生の HTTP リクエストを書くよりも、型安全かつ公式にメンテナンスされたストリーミングインターフェース (`GenerateContentStream`) を利用できるため、コードの堅牢性が向上します。

### 3. コンテナ設計 (Dockerfile)
- **決定**: **マルチステージビルド (Multi-stage Build)** を採用する。
  - ビルド環境: `golang:1.26-alpine`
  - 実行環境: `alpine`
- **理由**: ECS (Fargate) での稼働を見据え、ビルドツールのない軽量な実行コンテナにします。また、Gemini API と HTTPS 経由でセキュアに通信するために、実行環境コンテナに `ca-certificates` をインストールしてルート証明書を保持させます。

### 4. リアルタイム通信方式の選定 (SSE vs WebSocket)
- **決定**: **Server-Sent Events (SSE)** を採用する。
  - **理由**: 今回のユースケース（AIの返答を単方向にストリーミングする）に対して、WebSocket は双方向でオーバーヘッドが大きく、ECS上の ALB (Application Load Balancer) や API Gateway で接続を維持する際のインフラ構成・タイムアウト設定が複雑になります。SSE は標準的な HTTP/1.1 および HTTP/2 の上で動作するため、ロードバランサーを介した負荷分散や接続制御がはるかにシンプルです。

## 影響と考慮事項 (Consequences)

### メリット
- **ECSへの即時移植性**: Dockerfile や環境変数による設定ファイル（`.env` 等）をそのまま ECS のタスク定義 (Task Definition) に移行できます。
- **軽量・高速スケーリング**: コンテナサイズが小さいため、ECS上でのコールドスタート（コンテナ起動時間）が短縮され、急激なトラフィック増に対するオートスケーリングが迅速になります。

### 注意事項・トレードオフ
- **ロードバランサー (ALB) のバッファリング対策**: ECS/ALB を経由する際、プロキシがレスポンスをキャッシュ（バッファリング）してしまうと、SSE のストリーミングがブラウザに即時届かなくなります。バックエンド側で `X-Accel-Buffering: no` 等のヘッダーを出力するか、バッファリングを無効にする設定が必要です。
- **タイムアウト設定**: SSE は接続時間が長くなるため、ECS や ALB の「アイドルタイムアウト」によって切断されないよう、定期的なキープアライブ（空データ）の送信やクライアント側での自動再接続 (retry) ロジックが必要です。
